# Repo Patterns

Decision-making map for LLM agents: how apps are built, tested, deployed, and documented in this repo — which existing pattern to copy. For infra operations, secrets, data/persistence, local dev environment, and auth, see [repo-operations.md](./repo-operations.md).

## App Development

- **New API service** → `apps/api/*`, Fastify via `@vigilant-broccoli/fastify`, deployed to fly.io. Read `fly-service-pattern.md` first and copy one of its reference apps (build shape × image delivery).
- **New static UI app** → `apps/ui/*`, Vite + React, deployed to Cloudflare Pages — read `cloudflare-pages-deploy-pattern.md` first (wrangler target trio, per-env config, custom domains via Terraform). App-side requirements (cards, i18n): `ui-app-pattern.md`.
- **Next.js apps** are exceptions, each with its own deploy: `hearth`/`findme`/`whiteboard`/`employee-handler-ui` (Vercel — read `vercel-deploy-pattern.md` first), `vb-manager-next` (PM2 on the VM, no nx `deploy` target).
- **Shared code** → `libs/@vigilant-broccoli/*`. Check before writing new code: `common-js` (HTTP consts), `common-node` (`getEnvironmentVariable`, http utils), `react-lib` (components, `createI18n`), `fastify`/`express`/`next-lib`.
- **Local serve with secrets**: the `serve` target wraps `serve:no-vault` with `scripts/fetch-secrets.ts` + `NODE_EXTRA_CA_CERTS=./scripts/vault-ca.crt`. Top-level module init must never require live secrets or network (rule and rationale in `fly-service-pattern.md`).

## Deployment

- Deployment is **nx-target-driven**: an app is deployable iff its `project.json` defines a `deploy` target. `deploy.yml` runs the affected `deploy` targets on every push to main touching `projects/nx-workspace/**`; pushes to the `production` branch run every `deploy:production` target (not affected-based — `nx-set-shas` tracks last-successful runs on main, so there is no reliable affected base on `production`). The pipeline is encoded in `dependsOn` (typical fly chain: `build → lint → prune → smoke → deploy:secrets → deploy`).
- Destinations in use: fly.io (`apps/api/*`), Cloudflare Pages (static UIs), Vercel (Next.js apps), GitHub Pages (`deploy-github-pages` targets + `pages-index/`), npm (`publish-package` targets), Sanity (`cloud8skate-sanity`, `sanity deploy`), PM2-on-VM (`vb-manager-next`). Per-destination mechanics: `fly-service-pattern.md`, `vercel-deploy-pattern.md`, `cloudflare-pages-deploy-pattern.md`, `github-pages-deploy-pattern.md`.
- **Destination auto-creation**: a first deploy provisions its own target, so a new (or renamed) app/environment needs no manual console setup — fly apps via `deploy:secrets`, Cloudflare Pages projects via `ensure-cf-project`, Vercel projects via `deploy-vercel.ts` (mechanics in each destination's pattern doc). The only manual steps left are custom-domain cutovers (Terraform apply + freeing the domain from the old project).
- **Vercel domains are explicit records**, not derived from the project name — renaming a project does not move its domain, and `vercel deploy` never adds or removes domains (API commands in `vercel-deploy-pattern.md`).
- **npm package publishing**: a `libs/@vigilant-broccoli/*` lib publishes iff its `project.json` defines a `publish-package` target — `deploy-npm-packages` in `deploy.yml` gates on `nx show projects --withTarget=publish-package` and runs only for the production environment (pushes to the `production` branch or `environment: production` dispatches — never staging), publishing every such lib and skipping versions already on the registry. Having `publishConfig` in `package.json` alone does **not** get a lib published; several libs carry it as dead config with no wired target. Reference: `slack-workspace`, `slackbots`, `employee-handler`, `react-lib`. To add publishing to a lib:
  - `package.json`: add `publishConfig: { "access": "public" }` and a `repository` block.
  - `project.json`: add a `release.version` block (`currentVersionResolver: "git-tag"`, `manifestRootsToUpdate: ["dist/{projectRoot}"]`), a `publish-package` target (`nx:run-commands`, `dependsOn: ["build"]`, `cwd` set to the dist package root), and an `nx-release-publish` target (`packageRoot: "dist/{projectRoot}"`).
  - The `publish-package` command runs `scripts/resolve-workspace-deps.js` first (rewrites any `workspace:*` deps to their real published version — needed because most libs depend on `@vigilant-broccoli/common-js` via `workspace:*`), then checks `npm view <name>@<version>` and skips publishing if that version already exists, otherwise `npm publish --access public --provenance`.
  - CI auth: `deploy-npm-packages` pulls `NPM_TOKEN` from Vault (`kv/data/secrets NPM_TOKEN | NODE_AUTH_TOKEN`) via `.github/actions/vault-secrets`, exported as `NODE_AUTH_TOKEN` which `npm publish`/`setup-node` read automatically. The token must be an Automation or Granular Access Token (org-wide write, not scoped to specific existing packages) — a Classic token can trigger an interactive OTP prompt that fails non-interactively in CI, and a token scoped only to existing packages will 404 on any lib being published for the first time.
  - npm trusted publishing (OIDC, no stored token) is not usable for a lib's _first_ publish — npm requires the package to already exist on the registry before a trusted publisher can be attached to it. First publish of any new lib always goes through the `NPM_TOKEN` flow above; only later version bumps of already-published libs could move to OIDC if that's set up per-package on npmjs.com.
- **Secrets**: Vault is the source of truth. Declare per-service mappings in `projects/nx-workspace/scripts/secrets-mapping.config.ts`; `deploy:secrets` pushes them to fly. CI reaches Vault through `.github/actions/vault-secrets` (Cloudflare Access service token + GitHub OIDC JWT) — do not add plain GitHub repo secrets.
- **New deployable app checklist**: `deploy` (staging) + `deploy:production` targets (+ `smoke` and per-env fly configs for fly services), entry in `manual-deploy-app.yml` options, Upptime entries in `.upptimerc.yml` for each environment's public URL (or `ci-health-check` coverage if private), secrets mapping if needed, coverage in the environment-matrix `test-*` workflows (extend `test-security-api`'s service matrix or add a `test-*` workflow) if the app warrants post-deploy verification, `apps/ui/pages-index` UI Apps card for UI apps.
- **Environments**: everything that names a deployed instance is environment-prefixed with `staging-` (and `production-` when that environment is added) — fly app names and config filenames, Cloudflare Pages project names, Vercel project names (the `test-*` workflows derive per-env URLs from an environment matrix). Every deployable app carries a target pair — `deploy` (staging, runs on push to main) and `deploy:production` (runs on push to the `production` branch, or via the `deploy.yml`/`manual-deploy-app.yml` `environment` dispatch input). How each destination wires its environments (per-env fly configs, the `production-env` build configuration, per-env wrangler targets, `deploy-vercel.ts` env derivation) lives in its pattern doc; `journal` is deliberately single-environment (see `cloudflare-pages-deploy-pattern.md`). Nx project names stay environment-less — `nx deploy llm-service` deploys whatever environment its config points at. Custom domains are also environment-less: Terraform attaches each one to whichever environment's project should serve live traffic (today the `staging-*` projects; they move to `production-*` when that exists).

## CI Workflows

The filename prefix is the routing convention:

- `ci-*` — core CI: `ci-pr-check.yml` (PRs: `nx affected -t lint test build` + pre-commit), `ci-health-check.yml` (health probe: cron + dispatch + after deploy/rotate-secrets), `ci-rotate-secrets.yml` (secret rotation).
- `deploy.yml` — pushes to main deploy staging (affected-based); pushes to `production` deploy production (everything, via the `deploy:production` targets, plus npm publish); GitHub Pages deploys from main only. `workflow_dispatch` deploys everything and still takes an `environment` choice (`staging` default). `deploy-github-profile.yml` separately deploys `projects/github-profile/` to the profile repo on push to main.
- `cron-*` — scheduled; also dispatchable.
- `manual-*` — dispatch-only operations (deploy one app, kill services, replace code-server).
- `test-e2e-*` / `test-security-*` / `test-smoke-*` — checks against live deployed URLs; triggered by `workflow_run` after `deploy`/`rotate-secrets`, plus cron and dispatch. Workflows that target a deployed environment run a staging/production matrix (one workflow covers both environments, URLs derived from `matrix.environment`): cron tests both, dispatch takes an `environment` choice (`all` default), and `workflow_run` tests only the environment the completed run's branch deployed (`main` → staging, `production` → production; `ci-rotate-secrets` → staging). `test-*` workflows without an environment matrix check environment-less infrastructure (Vault, GCP, socket server, RabbitMQ, Cloudflare Access).
- `notify-complete.yml` — deploy-complete notification via the socket server (the deploy-start notification is a job inside `deploy.yml`).

## Testing

- Unit tests: `@nx/vite:test` (vitest), run via `nx affected` in CI.
- Pre-deploy gate: fly services must have a `smoke` target that boots the built dist with dummy env vars — the only thing that catches missing pruned deps (mechanics in `fly-service-pattern.md`).
- Post-deploy verification lives in GitHub workflows (`test-e2e-*`, `test-security-*`, running a staging/production environment matrix when they target a deployed environment) hitting live URLs, not in unit suites.
- Ongoing monitoring: Upptime for public URLs, `ci-health-check` for everything else. Both `cron-upptime.yml` (hourly up/down) and `cron-upptime-response-time.yml` (daily response-time) warm every fly.io URL (staging + production) with a parallel `curl` before invoking `upptime-monitor`, so a scaled-to-zero fly machine's cold start isn't misreported as downtime.

## Documentation

- App READMEs carry a `## Stack` bullet list.
