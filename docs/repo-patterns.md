# Repo Patterns

Decision-making map for LLM agents: how apps are built, tested, deployed, and documented in this repo. Binding conventions live in [CLAUDE.md](../CLAUDE.md) — this doc tells you which existing pattern to copy. For infra operations, secrets, data/persistence, local dev environment, and auth, see [repo-operations.md](./repo-operations.md).

## App Development

- **New API service** → `apps/api/*`, Fastify via `@vigilant-broccoli/fastify`, deployed to fly.io. Read [fly-service-pattern.md](../projects/nx-workspace/docs/deployment/fly-service-pattern.md) first and copy one of its reference apps (build shape × image delivery).
- **New static UI app** → `apps/ui/*`, Vite + React, deployed to Cloudflare Pages. Reference: `apps/ui/cloud-8-skate-angular` for the full staging+production setup (`ensure-cf-project`/`prune-deployments`/`deploy` wrangler trio per environment, plus the `production-env` build configuration); `apps/ui/journal` shows the minimal single-environment variant. Requires a card in `pages-index/ui.html` and i18n via `createI18n` (see CLAUDE.md). If the site gets a custom domain, Terraform owns the `cloudflare_pages_domain` attachment and its DNS record — one `cloudflare-<site>.tf` per site in `infrastructure/terraform/` (`cloudflare-cloud8skate.tf` is the plain pattern; `cloudflare-journal.tf` adds Access gating of the `*.pages.dev` aliases, required for private content).
- **Next.js apps** are exceptions, each with its own deploy: `hearth`/`findme`/`whiteboard`/`employee-handler-ui` (Vercel via `scripts/deploy-vercel.ts`, project name pinned in the `deploy` target's `VERCEL_PROJECT_ID`), `vb-manager-next` (PM2 on the VM, no nx `deploy` target).
- **Shared code** → `libs/@vigilant-broccoli/*`. Check before writing new code: `common-js` (HTTP consts), `common-node` (`getEnvironmentVariable`, http utils), `react-lib` (components, `createI18n`), `fastify`/`express`/`next-lib`.
- **Local serve with secrets**: the `serve` target wraps `serve:no-vault` with `scripts/fetch-secrets.ts` + `NODE_EXTRA_CA_CERTS=./scripts/vault-ca.crt`. Top-level module init must never require live secrets or network — defer to per-request.

## Deployment

- Deployment is **nx-target-driven**: an app is deployable iff its `project.json` defines a `deploy` target. `deploy.yml` runs the affected `deploy` targets on every push to main touching `projects/nx-workspace/**`. The pipeline is encoded in `dependsOn` (typical fly chain: `build → lint → prune → smoke → deploy:secrets → deploy`).
- Destinations in use: fly.io (`apps/api/*`), Cloudflare Pages (static UIs), Vercel (`hearth`), GitHub Pages (`deploy-github-pages` targets + `pages-index/`), npm (`publish-package` targets), Sanity (`cloud8skate-sanity`, `sanity deploy`), PM2-on-VM (`vb-manager-next`).
- **Destination auto-creation**: a first deploy provisions its own target, so a new (or renamed) app/environment needs no manual console setup. Fly apps are created by `deploy:secrets` (`deploy-flyio-secrets.ts` runs `flyctl apps create` when the app is missing; volumes declared in `[mounts]` auto-create on first `flyctl deploy`). Cloudflare Pages projects are created by each UI app's `ensure-cf-project` target. Vercel projects are created by `deploy-vercel.ts` (`ensureProjectExists`) — this requires `VERCEL_PROJECT_ID` to be the project _name_, not a `prj_` ID. The only manual steps left are custom-domain cutovers (Terraform apply + freeing the domain from the old project).
- **Vercel domains are explicit records, not derived from the project name**: renaming a project does not move its `<name>.vercel.app` domain, and `vercel deploy` never adds or removes domains — it only points the project's existing domains at the new deployment. Attach/detach via the API (`POST`/`DELETE /v9/projects/{idOrName}/domains`, `VERCEL_TOKEN` from Vault). When renaming a deployed instance, add the new `<name>.vercel.app` domain and remove the old one explicitly.
- **npm package publishing**: a `libs/@vigilant-broccoli/*` lib publishes iff its `project.json` defines a `publish-package` target — `deploy-npm-packages` in `deploy.yml` gates on `nx show projects --withTarget=publish-package` (affected-based on push, all of them on `workflow_dispatch`). Having `publishConfig` in `package.json` alone does **not** get a lib published; several libs carry it as dead config with no wired target. Reference: `slack-workspace`, `slackbots`, `employee-handler`, `react-lib`. To add publishing to a lib:
  - `package.json`: add `publishConfig: { "access": "public" }` and a `repository` block.
  - `project.json`: add a `release.version` block (`currentVersionResolver: "git-tag"`, `manifestRootsToUpdate: ["dist/{projectRoot}"]`), a `publish-package` target (`nx:run-commands`, `dependsOn: ["build"]`, `cwd` set to the dist package root), and an `nx-release-publish` target (`packageRoot: "dist/{projectRoot}"`).
  - The `publish-package` command runs `scripts/resolve-workspace-deps.js` first (rewrites any `workspace:*` deps to their real published version — needed because most libs depend on `@vigilant-broccoli/common-js` via `workspace:*`), then checks `npm view <name>@<version>` and skips publishing if that version already exists, otherwise `npm publish --access public --provenance`.
  - CI auth: `deploy-npm-packages` pulls `NPM_TOKEN` from Vault (`kv/data/secrets NPM_TOKEN | NODE_AUTH_TOKEN`) via `.github/actions/vault-secrets`, exported as `NODE_AUTH_TOKEN` which `npm publish`/`setup-node` read automatically. The token must be an Automation or Granular Access Token (org-wide write, not scoped to specific existing packages) — a Classic token can trigger an interactive OTP prompt that fails non-interactively in CI, and a token scoped only to existing packages will 404 on any lib being published for the first time.
  - npm trusted publishing (OIDC, no stored token) is not usable for a lib's _first_ publish — npm requires the package to already exist on the registry before a trusted publisher can be attached to it. First publish of any new lib always goes through the `NPM_TOKEN` flow above; only later version bumps of already-published libs could move to OIDC if that's set up per-package on npmjs.com.
- **Secrets**: Vault is the source of truth. Declare per-service mappings in `projects/nx-workspace/scripts/secrets-mapping.config.ts`; `deploy:secrets` pushes them to fly. CI reaches Vault through `.github/actions/vault-secrets` (Cloudflare Access service token + GitHub OIDC JWT) — do not add plain GitHub repo secrets.
- **New deployable app checklist**: `deploy` (staging) + `deploy:production` targets (+ `smoke` and per-env fly configs for fly services), entry in `manual-deploy-app.yml` options, Upptime entries in `.upptimerc.yml` for each environment's public URL (or `cron-health-check` coverage if private), secrets mapping if needed, `staging-test-*`/`production-test-*` workflows if the app warrants post-deploy verification, `apps/ui/pages-index` UI Apps card for UI apps.
- **Environments**: everything that names a deployed instance is environment-prefixed with `staging-` (and `production-` when that environment is added). For fly: the app name (`staging-vb-express`), its config filename (`deployment-configs/fly-configs/staging-<project>.toml`), `flyAppName` in `secrets-mapping.config.ts`, and the CI test workflows that hit it (`staging-test-*`). For Cloudflare Pages: the wrangler project name in `ensure-cf-project`/`prune-deployments`/`deploy` targets (`staging-journal`, `staging-cloud-8-skate-angular`, `staging-harryliu-dev-react`, `staging-harryliu-dev-angular`). For Vercel: the project name in each deploy target's `VERCEL_PROJECT_ID` (`staging-vb-hearth`, `staging-findme`, `staging-whiteboard`, `staging-employee-handler-ui`). Every deployable app carries a target pair — `deploy` (staging, runs on push to main) and `deploy:production` — and production deploys only via the `deploy.yml`/`manual-deploy-app.yml` `environment` dispatch input. Fly services use per-env fly configs (+ mirrored `deploy:secrets:production`) whose cross-service URLs point at same-env siblings. Static UI apps bake their fly URLs per env via a `production-env` build configuration (`fileReplacements` of `environment.ts` → `environment.production.ts`) and per-env wrangler targets (`ensure-cf-project:production`, `prune-deployments:production`). Vercel apps pass the environment to `deploy-vercel.ts`, which derives per-env project URLs and env vars (`VB_EXPRESS_URL`, `EMAIL_SERVICE_URL`, `NEXT_PUBLIC_APP_URL`). `journal` is deliberately single-environment: it's Access-gated private content deployed from Gitea by cron, and a second pages.dev alias would need Access coverage first. Nx project names stay environment-less — `nx deploy llm-service` deploys whatever environment its config points at. Custom domains are also environment-less: Terraform attaches each one to whichever environment's project should serve live traffic (today the `staging-*` projects; they move to `production-*` when that exists).

## CI Workflows

The filename prefix is the routing convention:

- `pr-check.yml` — PRs: `nx affected -t lint test build` + pre-commit.
- `deploy.yml` — pushes to main: npm publish, app deploys, GitHub Pages — all affected-based; `workflow_dispatch` deploys everything and takes an `environment` choice (`staging` default; `production` runs the `deploy:production` targets instead).
- `cron-*` — scheduled; must also have `workflow_dispatch`.
- `manual-*` — dispatch-only operations (deploy one app, kill services, replace code-server).
- `test-e2e-*` / `test-security-*` / `test-smoke-*` — checks against live deployed URLs; triggered by `workflow_run` after `deploy`/`rotate-secrets`, plus weekly cron and dispatch. Workflows that target a deployed environment carry that environment as a prefix on both the filename and the `name:` field (`staging-test-e2e-llm.yml` / `production-test-e2e-llm.yml`); unprefixed `test-*` workflows check environment-less infrastructure (Vault, GCP, socket server, RabbitMQ, Cloudflare Access). `production-test-*` workflows omit the `workflow_run` trigger — production deploys are manual dispatches, so they run on cron and dispatch only.
- `notify-*` — deploy start/complete notifications via the socket server.

Pin action versions to SHAs with a version comment; keep README badges alphabetical and in sync with workflows (see CLAUDE.md).

## Testing

- Unit tests: `@nx/vite:test` (vitest), run via `nx affected` in CI. Do not add tests unless asked (CLAUDE.md).
- Pre-deploy gate: fly services must have a `smoke` target (`scripts/shell/smoke-dist.sh`) that boots the built dist with dummy env vars — the only thing that catches missing pruned deps.
- Post-deploy verification lives in GitHub workflows (`test-e2e-*`, `test-security-*`, environment-prefixed as `staging-test-*` / `production-test-*` when they target a deployed environment) hitting live URLs, not in unit suites.
- Ongoing monitoring: Upptime for public URLs, `cron-health-check` for everything else.

## Documentation
